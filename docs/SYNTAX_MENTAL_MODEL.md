# SeedLang 语法心智模型：为什么有人觉得「杂」、怎么读

面向读者：觉得「显式隐式搅在一起、同一类东西写法不统一」的评论者，以及要在 Issue/文章里**对外解释**的维护者。规范细节仍以 **[LANGUAGE_SPEC_REFACTOR_DRAFT.md](LANGUAGE_SPEC_REFACTOR_DRAFT.md)** 为准。

## 1. 大家在抱怨的「杂」通常指什么

| 现象 | 说明 |
|------|------|
| 空格 / 逗号都能分隔 | 数组、对象、函数参数与实参：**空格与逗号都是合法分隔**（规范 §4.2、§4.3）。 |
| 运算符「两套脸」 | 逻辑运算常见 **符号** 与 **关键字** 并存（规范 §4.5，如 `&&` 与 `and`）。 |
| 多种 `for` 脸孔 | 例如 **C 风格** `for (i = 0; …)` 与 **迭代** `for item in arr`（规范 §4.4）。 |
| 没有单独的 `let` | 顶层/块里 **`name = value` 即绑定**，对习惯「声明关键字一眼认出作用域」的读者会显得「隐式」。 |

若读者默认「一门语言只该有一种体面写法」，就会把上面看成「多套语法硬缝在一起」——这在**观感**上成立，但在**语义**上多数是**同一规则下的可选表层**，不是两套互不相干的语言。

## 2. 对外口径（设计取舍，非道歉）

- **双轨表层**是刻意的：**偏极简 / AI 提示友好**（空格分隔）与 **偏 JS 迁移可读**（可选逗号、熟悉的操作符形状）并行，降低「从零学一套全新标点律」的摩擦。
- **教程与 AI 文档**（如 `AI_QUICK_START.md`）会**固定选一种写法**，避免初学者同时看到两种；**解析器**仍按规范接受另一种——避免把「教学默认」当成「唯一合法语法」。
- **实验特性**（类型层、部分异步边界等）在规范里单独标 `Experimental`；和「核心双轨」不是一类问题——前者是未完成/不稳定，后者是兼容层。

## 3. 推荐给贡献者与示例代码的「单一写法」

用来**减少杂感**（非强制，团队可自行约定）：

1. **列表与调用**：默认 **空格分隔**；行变长或从 JS 粘贴时再上逗号。
2. **逻辑**：项目内统一关键字 **`and` / `or` / `not`**，或统一符号 **`&&` / `||` / `!`**，不要同一文件混用两套。
3. **循环**：遍历集合优先 `for x in xs`；要数索引时，模块内固定用 **`while` 计数** 或 **C 风格 `for`** 之一，避免三种形态轮换出现。
4. **心智锚点**：把 SeedLang 想成 **「JS 语义 + 可选的空格优先表层」**，而不是「又学一门与 JS 无关的语言」。

## 4.「看起来不同、语义相同」速查

| 写法 A | 写法 B | 关系 |
|--------|--------|------|
| `[1 2 3]` | `[1, 2, 3]` | 等价（逗号可选） |
| `fn f(a b)` | `fn f(a, b)` | 等价 |
| `f(1 2)` | `f(1, 2)` | 等价 |
| `a && b` | `a and b` | 同类逻辑运算的不同字面（见规范 §4.5） |

## 5. 和「宏 / 卫生」那类真·语义分叉区分

宏形参代入、部分卫生等属于**另一类**话题（见规范 §7.4）；不要与「逗号可有可无」这类**纯表层**混谈，否则读者会觉得「到处都在特殊规则」。

## 6. 若仍想从语言层面「变纯」

那是 **大版本 / 破坏性变更** 级别：收窄合法语法、加 `lint:seed` 规则默认收紧、或提供 `--strict-style` 等——超出「文档解释」范围；本页只解决**认知与对外叙事**。
